Pare de construir no escuro.

O Design System da Corporal Class: tokens, tipografia Silka, componentes e acessibilidade prontos para você montar telas com consistência e parar de improvisar a cada tela nova. A decisão fica no token, não no achismo. Clique em qualquer cor, valor (hex/rgba) ou bloco de código para copiar.

Elevação semântica

Papéis mapeados na escala de sombra: Elevado = elev-1 · Sobreposição = elev-3 · Modal = elev-4. Use o papel (`--elev-raised/overlay/modal`), não o número. Sombras usam camadas pretas no dark e matiz da marca no light.

Cantos aninhados

Elemento arredondado dentro de outro: os cantos devem ser concêntricos (mesmo centro de arco). Regra: raio interno = raio externo − padding, pra o espaçamento ficar uniforme inclusive nos cantos.

Faça · concêntrico
Externo r 24 · padding 16 → interno r 8 (24 − 16). Gap uniforme.
Não faça · mesmo raio
Interno r 24 (igual ao externo) → nos cantos o gap "estufa". Não concêntrico.

CSS · derive o interno com calc()
/* raio interno = raio externo − padding (nunca negativo) */ .card{ --r:24px; --pad:16px; border-radius:var(--r); padding:var(--pad); } .card > .inner{ border-radius:max(0px, calc(var(--r) - var(--pad))); } /* 24 - 16 = 8 */

Padding ≥ raio → interno vira 0 (reto). Borda/stroke inverte: externo = interno + largura da borda. Aninhamento profundo: subtraia o padding de cada nível. Pill é exceção (não recalcula). Com corner smoothing (squircle iOS) a fórmula vira aproximação: iguale o smoothing em pai e filho.

Princípios e processo

Consistência por token, paridade entre claro e escuro, acessibilidade por padrão. Decida uma vez, reaplique em todo lugar.

Arquitetura de tokens (3 camadas)

Tokens fluem em primitive → semantic/intent → component. Regra de ouro: o token semântico nunca sugere valor (--btn-bg, não --azul-500); o de componente referencia o semântico, nunca o primitivo direto. É o que torna theming e rebrand triviais.

CamadaExemploPapel
primitive#hex · 16pxValor cru, sem contexto.
semantic--accent · --space-4Intenção / uso.
component--btn-bgAplicação no componente.
Princípios de movimento

Anime com propósito (feedback, orientação, continuidade), nunca decorativo. Durações nos tokens (--motion-fast .15 · --motion .2 · --motion-slow .4), curva --ease. Respeite prefers-reduced-motion (já no reset). Não anime: deslocamento de layout nem texto de leitura.

Versionamento e contribuição

SemVer (MAJOR.MINOR.PATCH) + CHANGELOG.md + política de depreciação. Contribuição: regra das 3 equipes (só vira padrão o que serve 3+ usos) + revisão antes de virar componente. Ver CONTRIBUTING.md no pacote.

Faça / Não faça
Faça
  • Consumir tokens (nunca hex solto).
  • Escuro + claro em paridade, contraste WCAG AA.
  • Ícone + texto nos estados (cor nunca sozinha).
  • Foco visível em tudo que é focável.
Não faça
  • Cor / raio / sombra fora dos tokens.
  • Texto claro sem contraste no light.
  • Animar sem propósito ou ignorar reduced-motion.
  • Botão <a> sem href.

Separação de domínio

Cada marca só fala do próprio domínio. Os termos específicos ficam em glossario-marca.md (um por marca) e nunca cruzam para outro design system. Na dúvida, erre para o neutro (corpo de texto, preenchimento de cor, token, cartão), nunca para o domínio de outra marca.
